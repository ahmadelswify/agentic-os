# STAR Stories Framework

## What Makes a Great Interview Story

A great story is not a chronological retelling of events. It's a structured argument for why you're the right person for the job.

## The Algorithm Label System

Before you tell any story, categorize the core action:

| Label | Meaning | Example |
|---|---|---|
| **Question** | You challenged an assumption | "I asked why we were doing X when the data showed Y" |
| **Delete** | You removed unnecessary work | "I eliminated a manual review step that added no value" |
| **Simplify** | You reduced complexity | "I consolidated 5 tools into 1 unified workflow" |
| **Accelerate** | You made something faster | "I automated the triage process, cutting response time 80%" |
| **Automate** | You removed human effort entirely | "I built a self-service system that handled 60% of requests" |

**Lead with the label.** Instead of "Well, there was this problem with our support tickets..." say "This is a Simplify story. We had five disconnected tools for one workflow, and I consolidated them into a single platform."

## Story Structure

### 1. Situation (2-3 sentences)
Set the stage. What was the context? What constraints existed? Make it vivid but brief.

### 2. Task (1 sentence)
What was YOUR specific responsibility? Not the team's goal. Yours.

### 3. Action (the bulk)
What did YOU do? Step by step. Use "I" not "we."
- What did you investigate?
- What did you decide?
- What did you build/change/implement?
- Who did you convince or align?

### 4. Result (1-2 sentences)
End with a number. Always. If you don't have an exact metric, estimate conservatively and say so.

## The 90-Second Rule

Your executive summary of any story should take 90 seconds or less. Time yourself. If you're over 90 seconds, you're going too deep too early.

Structure for the 90-second version:
- **10 seconds:** Algorithm label + one-sentence context
- **20 seconds:** The problem and why it mattered
- **40 seconds:** What you did (headline actions only)
- **20 seconds:** The result with metrics

Go deeper only when the interviewer probes. That's the signal they want more detail.

## Probe Answers

For each story, prepare answers to these common follow-ups:

- "What was the hardest part?"
- "What would you do differently?"
- "How did you get buy-in?"
- "What did you learn?"
- "How did you measure success?"
- "What happened after you left / moved on?"

Write these out. Don't wing them.

## Role-Type Variations

The same story can emphasize different aspects depending on the role:

| If applying for... | Emphasize... |
|---|---|
| Engineering/Technical | The technical decisions, architecture, tooling |
| Product Management | The user impact, prioritization, stakeholder alignment |
| Operations | The process improvement, efficiency gains, scalability |
| Customer Success | The customer impact, relationship management, retention |
| Sales/SE | The revenue impact, partner relationships, deal mechanics |

## Maintaining Your Stories

- **After each interview:** Update the outcome log. What version did you tell? How did it land? What follow-ups came?
- **After each outcome:** If the story contributed to an offer, mark it as proven. If it fell flat, note why and adjust.
- **Quarterly:** Review all stories. Archive ones that are stale. Refresh metrics if they've changed.
